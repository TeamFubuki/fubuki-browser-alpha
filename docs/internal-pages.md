# Internal Pages

Internal pages are SolidJS components in `ui/src/pages/InternalPages.tsx`. The
native scheme handler keeps each `fubuki://` route and serves Vite's dedicated
`internal.html` entry point plus its static assets.

Current pages:

- `fubuki://newtab/`
- `fubuki://history/`
- `fubuki://bookmarks/`
- `fubuki://downloads/`
- `fubuki://settings/`
- `fubuki://debug/`

The internal origin does not receive the Frost Protocol bridge: only `fubuki://app/` can use it.
Each internal origin reads a same-origin `/data.json` projection generated by the native scheme
handler from FrostEngine's SQLite store. This endpoint is read-only and returns only the data needed
by that page. Mutations use a separate capability-limited internal action channel. The native client
checks both the source page and the action key, applies the change through FrostEngine, and responds
without navigating or reloading the page. This preserves scroll, focus, and in-progress input.

The legacy `fubuki://settings/set` form route remains available for compatibility, but destructive
GET navigation is rejected. New SolidJS internal-page controls must use the internal action channel.

The UI supports persisted light, dark, and system appearance settings, English and Japanese labels,
keyboard focus states, reduced motion, responsive settings navigation, searchable history and
settings, and loading/error/empty states. In Vite development, append `?page=settings` (or another
internal page host) to `internal.html` to preview a page with non-sensitive sample data.

Keep these pages simple: searchable lists, useful empty states, and direct actions. Avoid controls
without backing native behavior. New state must remain owned by FrostEngine; `/data.json` is only a
read projection.
